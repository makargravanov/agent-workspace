# UI Structure: agent-workspace

## 1. Назначение

Этот документ фиксирует общую структуру пользовательского интерфейса `agent-workspace`.

Здесь описываются:

- основные контуры интерфейса;
- информационная архитектура;
- рекомендуемая route-структура;
- базовые принципы организации экранов;
- разделение между рабочим UI, workspace-админкой и глобальным операторским контуром.

Этот документ пока не является:

- визуальным дизайн-гайдом;
- спецификацией компонентной библиотеки;
- пиксельным описанием экранов;
- детальной API-спецификацией frontend-backend взаимодействия.

## 2. Принципы

- Один основной web-клиент, а не набор разрозненных frontend-приложений.
- `Workspace` является верхней пользовательской границей, `Project` — основной рабочей границей.
- Большая часть повседневной работы происходит внутри конкретного `Project`.
- Workspace-настройки и операторские настройки не должны смешиваться с повседневной рабочей навигацией.
- UI должен использовать бизнес-термины, а не внутренние названия сущностей там, где это возможно.
- Поиск и быстрый переход должны быть доступны из любой рабочей точки.
- Для задач, страниц базы знаний и заметок базовым паттерном является список или дерево плюс контекстная детальная область.
- Страницы документации и файловые вложения не должны смешиваться в одну пользовательскую сущность.
- Desktop — основной режим для управления, администрирования и работы с документацией; mobile должен покрывать базовые сценарии чтения, поиска, заметок и простых обновлений.

## 3. Основные контуры UI

### 3.1 Публичный и auth-контур

Назначение:

- вход пользователя;
- callback внешнего auth provider;
- локальный `dev-auth` режим только для development.

Типовые маршруты:

- `/login`
- `/auth/callback`
- `/dev-login` только для development

### 3.2 Глобальный пользовательский контур

Назначение:

- выбор workspace;
- создание workspace;
- переход по приглашению;
- попадание в последний активный workspace.

Типовые маршруты:

- `/workspaces`
- `/workspaces/new`
- `/invites/:inviteToken`

Этот контур должен быть минимальным и быстрым. Его задача — привести пользователя в нужный workspace, а не становиться отдельной рабочей средой.

### 3.3 Workspace-контур

Назначение:

- показать список проектов workspace;
- дать общий обзор workspace;
- предоставить доступ к workspace settings;
- обеспечить переключение между проектами.

Типовые маршруты:

- `/w/:workspaceSlug`
- `/w/:workspaceSlug/projects`
- `/w/:workspaceSlug/settings/...`

Этот слой нужен как контейнер. Основная ежедневная работа пользователя должна происходить на уровне project-контекста.

### 3.4 Project-контур

Назначение:

- основная рабочая область для задач, групп задач, базы знаний, заметок, поиска и активности;
- чтение и обновление проектного контекста;
- навигация по связанным сущностям.

Типовые маршруты:

- `/w/:workspaceSlug/p/:projectSlug/overview`
- `/w/:workspaceSlug/p/:projectSlug/tasks`
- `/w/:workspaceSlug/p/:projectSlug/groups`
- `/w/:workspaceSlug/p/:projectSlug/docs`
- `/w/:workspaceSlug/p/:projectSlug/docs/*documentPath`
- `/w/:workspaceSlug/p/:projectSlug/notes`
- `/w/:workspaceSlug/p/:projectSlug/search`
- `/w/:workspaceSlug/p/:projectSlug/activity`

### 3.5 Workspace-админка

Назначение:

- управление участниками workspace;
- управление агентами и их credentials;
- управление интеграциями;
- просмотр workspace-аудита.

Типовые маршруты:

- `/w/:workspaceSlug/settings/overview`
- `/w/:workspaceSlug/settings/members`
- `/w/:workspaceSlug/settings/agents`
- `/w/:workspaceSlug/settings/agent-credentials`
- `/w/:workspaceSlug/settings/integrations`
- `/w/:workspaceSlug/settings/audit`

Этот контур остается частью основного web-приложения, но должен быть визуально и логически отделен от project-работы.

### 3.6 Глобальный операторский контур

Назначение:

- управление всей инсталляцией системы;
- просмотр состояния workspace и системных компонентов;
- выполнение platform-level операций.

Типовые маршруты:

- `/operator`
- `/operator/workspaces`
- `/operator/operators`
- `/operator/audit`
- `/operator/system-health`
- `/operator/integrations`

Этот контур не должен смешиваться с ролями `WorkspaceMember`. Он предназначен для внутреннего оператора платформы.

## 4. Рекомендуемая route-структура

```text
/
  /login
  /auth/callback
  /dev-login

  /workspaces
  /workspaces/new
  /invites/:inviteToken

  /w/:workspaceSlug
    /projects
    /settings
      /overview
      /members
      /agents
      /agent-credentials
      /integrations
      /audit

    /p/:projectSlug
      /overview
      /tasks
      /tasks/:taskId
      /groups
      /groups/:groupId
      /docs
      /docs/*documentPath
      /notes
      /notes/:noteId
      /search
      /activity

  /operator
    /workspaces
    /operators
    /audit
    /system-health
    /integrations
```

## 5. Основные shell-структуры

### 5.1 Глобальный shell

Используется после входа, но до погружения в конкретный project.

Основные элементы:

- профиль пользователя;
- переключение workspace;
- быстрый переход в последний workspace;
- кнопка создания workspace.

### 5.2 Workspace shell

Основные элементы:

- переключатель workspace;
- список или переключатель проектов;
- переход в workspace settings для `owner`;
- краткая workspace-навигация.

Этот shell не должен быть перегружен. Его задача — довести пользователя до project-shell.

### 5.3 Project shell

Основные элементы:

- глобальная шапка с workspace switcher, project switcher, поиском, быстрым созданием и пользовательским меню;
- боковая навигация по основным разделам проекта;
- центральная рабочая область;
- правая контекстная панель или drawer для деталей, связей, активности и быстрых действий.

Рекомендуемые разделы навигации проекта:

- `Overview`
- `Tasks`
- `Groups`
- `Docs`
- `Notes`
- `Search`
- `Activity`

### 5.4 Workspace settings shell

Основные элементы:

- отдельный layout с явным признаком, что пользователь находится в настройках workspace;
- левая или верхняя секционная навигация;
- акцент на таблицах, формах, статусах и подтверждениях критичных действий.

### 5.5 Operator shell

Основные элементы:

- отдельный namespace и отдельный layout;
- визуально более строгий и явно внутренний режим;
- доступ только по operator-policy, а не по ролям workspace;
- концентрация на таблицах, фильтрах, health-статусах, журналах и аварийных действиях.

## 6. Экранные группы

### 6.1 Project Overview

Что должно быть видно в общих чертах:

- активные задачи и задачи с блокировками;
- ключевые группы задач;
- последние документы и заметки;
- недавняя активность людей, агентов и интеграций;
- важные GitHub-события, если интеграция подключена.

### 6.2 Tasks

Базовый паттерн:

- список задач с фильтрами;
- статус, приоритет, ответственный, блокировки;
- детальная панель задачи;
- быстрые действия без ухода со страницы.

Для MVP я бы исходил из list-first подхода. Kanban и более сложные представления можно добавить позже.

Важно: list-first в MVP не означает list-only на уровне данных. В перспективе один и тот же task model должен поддерживать list, board, hierarchy, timeline, calendar и dependency-oriented представления без дублирования сущностей.

### 6.3 Groups

UI не должен показывать внутренний термин `TaskGroup` напрямую. Лучше использовать пользовательский термин `Groups` или явную подпись `Initiatives and Epics`, если в конкретном интерфейсе это понятнее.

Что важно показывать:

- тип группы;
- статус;
- прогресс;
- дочерние задачи;
- связанные документы и заметки.

### 6.4 Docs / Knowledge Base

Этот раздел должен восприниматься как проектная база знаний, а не как файловый менеджер.

Базовый паттерн:

- дерево или каталог страниц слева;
- просмотр markdown-страницы в центральной области;
- режим редактирования с markdown-вёрсткой;
- встраивание изображений, схем, ссылок на задачи, заметки, внешние ресурсы и файловые assets;
- история версий и metadata страницы на следующем слое детализации.

Что важно поддержать в общих чертах:

- внутренние ссылки между страницами;
- древовидную структуру документации;
- вставку изображений;
- вставку схем, например Mermaid;
- ссылки на прикрепленные файлы и внешние документы.

### 6.5 Files / Assets

Файлы и изображения не должны быть главным пользовательским разделом наравне с задачами. В MVP это вспомогательный контур для документации, заметок и ссылок.

Базовый паттерн:

- загрузка файла из редактора документации или формы заметки;
- просмотр metadata файла;
- безопасное копирование ссылки на файл;
- использование файла как вложения или встроенного ресурса.

Отдельный top-level раздел `Files` не обязателен в MVP. Достаточно встроенного asset-picker и точек входа из Docs, Notes и контекстных форм.

### 6.6 Notes

Базовый паттерн:

- быстрый поток заметок;
- быстрый ввод новой заметки;
- фильтрация по автору, типу и проекту;
- привязка к задачам, группам, агентным сессиям, страницам документации и файлам.

### 6.7 Search

Поиск должен быть глобально доступен из project-shell и вести на отдельную результатную страницу.

Что должно быть в результате:

- смешанная выдача по задачам, группам, страницам документации, заметкам и файловым assets;
- объяснение, почему объект найден;
- быстрый переход в детальный контекст;
- фильтрация по типу, статусу и проектным параметрам.

### 6.8 Activity

Это пользовательский, а не административный журнал.

Он должен показывать:

- проектные изменения;
- активность агентов;
- изменения задач, страниц документации и файлов;
- связанные интеграционные события.

Это не замена security-аудиту, а рабочая лента событий проекта.

## 7. Правила терминологии в UI

- `WorkspaceMember` в UI отображается как `Member`.
- `Document` в UI отображается как `Doc` или `Page`, а не как абстрактный `Document`.
- `Asset` в UI отображается как `File`, `Image` или `Attachment`, в зависимости от контекста.
- `AgentCredential` в UI отображается как `Agent Key` или `Agent Credential`, в зависимости от контекста.
- `TaskGroup` в UI не должен светиться как внутренний термин. Предпочтительно использовать `Group`, `Initiative` или `Epic`.
- `AuditEvent` в пользовательском UI лучше показывать как `Audit` или `Security Events`, в зависимости от раздела.
- `AgentSession` в пользовательском UI лучше показывать как `Agent Session` или `Agent Run`, если это окажется понятнее для команды.

## 8. Базовая матрица доступа на уровне UI

- `viewer`: чтение project-контента без административных функций;
- `editor`: рабочие операции с задачами, заметками, страницами документации и связанными сущностями;
- `owner`: все возможности `editor` плюс workspace settings;
- `system operator`: отдельный внутренний контур уровня всей инсталляции.

Важно: скрытие раздела в UI не является защитой. Все policy checks должны выполняться на сервере.

## 9. Адаптивность

- Desktop является основным режимом для управления, базы знаний и администрирования.
- Mobile должен поддерживать просмотр, поиск, быстрые заметки и базовые обновления задач.
- Split-view на мобильных устройствах должен схлопываться в последовательную навигацию: список, затем детальная страница.
- Табличные административные разделы на mobile могут переходить в карточный список с отдельным экраном деталей.

## 10. Что не фиксируется пока

- точный визуальный стиль;
- финальная дизайн-система;
- набор низкоуровневых UI-компонентов;
- board-view для задач;
- отдельный operator frontend как самостоятельное приложение.
