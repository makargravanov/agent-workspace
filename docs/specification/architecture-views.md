# Architecture Views

## 1. Назначение

Этот документ фиксирует подход к архитектурным представлениям `agent-workspace`.

Цель — описывать архитектуру последовательно и понятно для разных аудиторий: продукта, разработки, интеграций и эксплуатации.

## 2. Рекомендуемый подход

Для архитектурной документации проекта следует использовать подход C4 как общую модель представлений.

Смысл этого выбора:

- C4 дает простую иерархию уровней;
- помогает не смешивать разные уровни детализации в одной диаграмме;
- подходит для modular monolith, API, web, MCP и интеграционных контуров;
- понятен и технарям, и нетехническим стейкхолдерам.

На уровне практики C4 здесь используется как модель описания, а не как жесткая привязка к одному инструменту.

## 3. Какие представления нужны сейчас

### 3.1 Level 1: System Context

Нужно показать:

- людей и их роли;
- систему `agent-workspace` как единое целое;
- GitHub и другие внешние системы;
- agent IDE / CLI и MCP-контур;
- границы взаимодействия между человеком, агентом и системой.

Аудитория:

- продукт;
- архитектура;
- разработка;
- стейкхолдеры, которым нужен общий обзор.

### 3.2 Level 2: Container

Нужно показать:

- web-клиент;
- API;
- PostgreSQL;
- object storage для assets;
- MCP bridge;
- внешние integration providers.

Аудитория:

- архитекторы;
- backend/frontend разработка;
- DevOps.

### 3.3 Level 3: Component

Нужно делать не сразу, а после стабилизации модульных границ API.

Первый кандидат на component-диаграмму:

- `services/api`

Вероятные блоки для отображения:

- workspace core;
- task management;
- knowledge base;
- auth/access;
- agent sessions;
- search indexing;
- github integration;
- asset storage adapter.

### 3.4 Level 4: Code

Для проекта сейчас не обязателен.

Если когда-то понадобится, его лучше строить автоматически из кода или использовать только для локально сложных участков, а не как основной формат документации.

## 4. Дополнительные представления

Помимо базовых уровней C4 здесь полезны:

- deployment view для локального Docker и production/self-hosted профилей;
- dynamic view для flows входа, agent session, asset upload, GitHub sync;
- security-oriented view для human auth, agent credentials, MCP и operator contour.

## 5. Инструментальный подход

На старте стоит придерживаться text-based подхода и хранить диаграммы в репозитории рядом с документацией.

Рекомендуемый прагматичный вариант:

- использовать markdown-документы в `docs/`;
- встраивать диаграммы в виде Mermaid там, где это достаточно выразительно;
- следовать semantics C4 даже там, где используется не специализированный C4-инструмент;
- если архитектурная документация усложнится и понадобится единый executable source of truth, позже перейти на Structurizr DSL или аналогичный инструмент.

## 6. Что стоит нарисовать в ближайшую очередь

1. System Context diagram.
2. Container diagram.
3. Deployment view для локального профиля.
4. Dynamic view для human auth, agent auth и asset upload.
