# Анализ примера ТЗ

## 1. Что удалось извлечь

Сравнивались два источника одного и того же примера ТЗ:

- `ТЗ (пример).pdf`
- `ТЗ (пример).docx`

Вывод:

- PDF читается, но в извлечении появляются артефакты верстки: разрывы страниц, лишние пробелы, сбитые списки и менее чистая структура.
- DOCX извлекается заметно лучше: заголовки, нумерация, абзацы и перечни читаются стабильнее.
- Для дальнейшей работы над нашим ТЗ в качестве основного структурного ориентира стоит использовать именно DOCX-версию примера.
- PDF полезен как дополнительный источник для визуальной проверки порядка разделов и общего вида итогового документа.

## 2. Выявленная структура ТЗ из примера

По DOCX-версии пример ТЗ имеет следующую общую структуру:

- титульный лист;
- `1 Введение`;
- `2 Назначение и область применения`;
- `3 Цель разработки`;
- `4 Термины и сокращения`;
- `5 Требования к программе`.

Внутри раздела `5 Требования к программе` в примере есть такие подразделы:

- `5.1 Функциональные требования`;
- `5.2 Требования к интерфейсу`;
- `5.3 Требования к реализации`;
- `5.4 Требования к надежности`;
- `5.5 Требования к тестированию`;
- `5.6 Требования к установке`;
- `5.7 Требования к сопровождению`;
- `5.8 Требования к безопасности`;
- `5.9 Требования к документации`.

Это выглядит как хорошая базовая рамка для нашего вузовского ТЗ.

## 3. Что это значит для нашего проекта

Для `agent-workspace` эта структура подходит, но содержание разделов должно быть иным.

Что можно брать почти напрямую как форму:

- титульный лист;
- введение;
- назначение и область применения;
- цель разработки;
- термины и сокращения;
- блок требований;
- требования к безопасности;
- требования к документации.

Что придется существенно адаптировать:

- требования к интерфейсу: у нас web, workspace admin, operator contour, knowledge base, task UI;
- требования к реализации: у нас Rust API, React frontend, PostgreSQL, asset storage, MCP bridge, Docker;
- требования к надежности: у нас нет смысла копировать требования под выделенные серверы из примера;
- требования к установке: вместо USB и WinWord у нас будут Docker, self-hosted профиль, dev-профиль, возможно LaTeX/PDF как итоговый формат отчета;
- требования к сопровождению: надо формулировать под наш продукт, а не под шаблонный корпоративный кейс.

Что у нас уже есть как сильная база для будущего ТЗ:

- продуктовая спецификация проекта;
- ER-модель и словари значений;
- структура UI;
- решения по auth, admin, operator contour;
- решение по storage для assets;
- архитектурные представления через C4.

## 4. Как использовать LaTeX-шаблон

Найденный шаблон `novsu-thesis-template-main` можно использовать как основу оформления, но не как готовую структуру содержания.

Судя по `main.tex`, шаблон сейчас ориентирован на дипломный документ и включает:

- титульный лист;
- аннотацию;
- содержание;
- введение;
- главы основного текста;
- заключение;
- список литературы;
- приложение.

Для ТЗ это означает:

- стиль и оформление можно переиспользовать;
- модульную структуру через `\include{...}` тоже можно переиспользовать;
- порядок и набор разделов надо заменить на структуру из примера ТЗ;
- аннотация, дипломные главы и часть стандартных разделов шаблона для ТЗ, скорее всего, не нужны в текущем виде.

Практически лучше сделать отдельную сборку под ТЗ, а не ломать дипломный `main.tex` напрямую.

Внутри раздела `5`:

## 6. Практический вывод

На текущем этапе:

- в качестве опорной структуры ТЗ лучше использовать DOCX-пример;
- PDF держать как визуальный контрольный образец;
- LaTeX-шаблон использовать как основу оформления, но не как источник структуры разделов;
- следующее рациональное действие — собрать черновой outline нашего ТЗ по структуре примера и разложить разделы по уже готовым спецификациям проекта.
