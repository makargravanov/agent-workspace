# ADR-0001: Bootstrap Architecture

## Status

Accepted

## Context

Нужно быстро запустить новую систему `agent-workspace`, сохранив при этом:

- низкий инфраструктурный оверхед;
- низкое потребление памяти;
- удобную локальную разработку;
- путь к дальнейшему масштабированию;
- пригодность для агентных интеграций через MCP.

На входе уже известны ключевые предпочтения:

- Rust на серверной стороне;
- React + TypeScript на фронтенде;
- контейнерное развертывание;
- возможная потребность в GitHub-интеграции;
- возможная потребность в семантическом поиске.

## Decision

### 1. Репозиторий

Используется монорепозиторий со следующими зонами ответственности:

- `services/api` — основной HTTP API и доменная логика;
- `tools/mcp-bridge` — отдельная MCP-утилита;
- `web` — фронтенд;
- `docs` — спецификации и архитектурные решения;
- `infra/docker` — инфраструктурные контейнерные артефакты.

### 2. Архитектурный стиль

Первая версия строится как модульный монолит.

Причина: доменные границы пока проектируются, и преждевременный переход к микросервисам принесет больше стоимости, чем пользы.

### 3. Хранилище данных

Основная СУБД — PostgreSQL.

Для семантического поиска закладывается `pgvector`, а не отдельная векторная база.

### 4. Интеграции

GitHub-интеграция проектируется как отдельный модуль внутри API.

### 5. Доступ агентов

Агенты подключаются не напрямую к базе и не напрямую к внутренним модулям, а через отдельную MCP-утилиту, которая использует API системы.

## Consequences

### Positive

- минимальный operational overhead на старте;
- простое локальное поднятие через Docker Compose;
- проще обеспечивать аудит и согласованность данных;
- база и семантический поиск живут в одном стеке;
- легко выделить модули в отдельные сервисы позже, если появятся реальные причины.

### Negative

- единый API остается точкой концентрации ответственности;
- потребуется дисциплина по модульным границам внутри монолита;
- при росте поисковой или интеграционной нагрузки позже может понадобиться вынос отдельных контуров.

## Revisit Criteria

К этому решению стоит вернуться, если:

- отдельный поисковый контур начнет масштабироваться независимо от core-домена;
- GitHub sync превратится в тяжелый самостоятельный пайплайн;
- MCP-контур потребует независимого жизненного цикла и деплоя;
- один контейнер API начнет мешать автономным поставкам разных продуктовых зон.
