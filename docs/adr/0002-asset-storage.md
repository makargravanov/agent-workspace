# ADR-0002: Asset Storage Abstraction

## Status

Accepted

## Context

В `agent-workspace` появляются два разных класса данных:

- структурированные доменные данные, metadata и связи, которые хорошо живут в PostgreSQL;
- бинарные файлы и изображения, которые используются как вложения и ресурсы базы знаний.

Нужно принять решение, где хранить файловые assets так, чтобы:

- локальная разработка и self-hosted запуск оставались простыми;
- hosted или production-развертывание не упиралось в локальную файловую систему одного контейнера;
- доменная модель не зависела от конкретного storage provider;
- модель доступа и безопасности оставалась управляемой из API.

## Decision

### 1. Разделение metadata и бинарного содержимого

Metadata asset-объектов хранятся в PostgreSQL.

Бинарное содержимое не хранится в PostgreSQL и выносится в отдельный storage backend.

### 2. Абстракция storage backend

В API вводится абстракция `AssetStorage` с единым интерфейсом для основных операций:

- подготовка загрузки;
- сохранение файла;
- получение файла;
- удаление файла;
- генерация безопасной ссылки на скачивание или чтение;
- проверка существования объекта.

### 3. Поддерживаемые backend-реализации

На первом этапе поддерживаются две реализации:

- `local` — локальная файловая система или Docker volume для development и простых self-hosted сценариев;
- `s3-compatible` — объектное хранилище с S3 API для production, hosted и более устойчивых self-hosted инсталляций.

### 4. Рекомендуемое использование по окружениям

- development по умолчанию использует `local` backend;
- self-hosted может использовать как `local`, так и `s3-compatible`, в зависимости от требований к надежности;
- production и hosted-инсталляции должны использовать `s3-compatible` backend.

### 5. Доступ к файлам

Assets по умолчанию считаются приватными.

Доступ к ним контролируется сервером приложения через policy checks.

Допустимы два механизма выдачи файла:

- proxy-download через API;
- серверная генерация short-lived signed URLs.

Конкретный механизм можно выбирать на уровне deployment-профиля, но логика авторизации всегда остается на стороне API.

## Consequences

### Positive

- локальная разработка остается простой и не требует обязательного внешнего object storage;
- production-архитектура не зависит от локального диска контейнера;
- смена backend storage не ломает доменную модель `Asset`;
- можно поддерживать self-hosted и hosted сценарии одной архитектурой;
- проще внедрить миграцию между storage backend-ами позже.

### Negative

- появляется дополнительный инфраструктурный контур по сравнению с хранением всего в одной БД;
- нужен lifecycle management для очистки orphaned assets;
- нужно синхронно учитывать backup/restore для БД и object storage;
- signed URL и proxy-download добавляют отдельные security и operational решения.

## Follow-up Notes

На следующем слое детализации стоит заранее учесть:

- ограничения по размеру файлов;
- allowlist MIME-типов и server-side content sniffing;
- hash файла для дедупликации или верификации целостности;
- lifecycle-статусы asset-объекта;
- стратегию удаления неиспользуемых файлов;
- резервное копирование и восстановление asset storage;
- возможное антивирусное или content-security сканирование, если система начнет принимать внешние пользовательские файлы в больших объемах.
